CODE-TO-DOCUMENTATION CONFORMANCE AUDIT
=======================================

Audit Goal
----------
Perform a file-by-file code-to-doc conformance review before rewriting the
documentation. The review standard used here was:

1. Read the current code file.
2. Compare existing documentation claims against the file's real behavior.
3. Consult bundled documentation and tests when a behavior boundary needed
   confirmation.
4. Run the bundled automated tests to verify behavior that was explicitly
   covered by the repository test suite.
5. Rewrite documentation only after the code/document comparison was complete.

Validation Actions Performed
----------------------------
- Extracted and reviewed the full repository from the uploaded ZIP.
- Inspected the source tree file by file.
- Verified module/function inventory across the Python files.
- Ran the bundled pytest suite for the two included test modules.
- Observed result after the latest revision: 48 tests passed, plus 1 opt-in live pipeline test skipped by default.

Command used during validation:
    python -m unittest -v

High-Level Findings
-------------------
1. The internal per-file Python docstrings were generally strong and mostly
   aligned with current behavior.
2. The top-level documentation files were improved compared with the prior ZIP
   but still had factual gaps and some over-broad wording.
3. The largest factual issue in the README was the dependency section: it named
   packages such as `requests` even though the current code uses `urllib` and
   `requirements.txt` currently lists only `beautifulsoup4>=4.12`.
4. The top-level docs under-described several confirmed UI and runtime features,
   including:
   - GPU/VRAM live readout in the main window
   - token/second meter behavior
   - session rename/delete/new workflow
   - exact settings fields present in the settings window
   - forced-search fallback logic in the Ollama client
5. The prior documentation change log contained generalized claims but not a
   defensible file-by-file conformance record. This audit file replaces that gap.

Confidence Statement
--------------------
The rewritten documentation is materially more defensible than the previous
version because it was produced after a file-by-file review and test run.
That said, the audit scope is still limited to:
- the code present in this ZIP
- the bundled tests
- behavior inferable from direct source inspection

It does not substitute for live end-to-end validation against the user's exact
Ollama instance, SearXNG instance, or GPU toolchain on their own machine.

Dedicated Search-Reranker Addendum
---------------------------------
This package was later revised again to add a dedicated reranker stage between
SearXNG search and the main answer model.

Changes confirmed from the current code:
- `services/reranker_service.py` was added as a new service-layer file.
- `services/ollama_client.py` now applies reranking to `search_web` results
  when the saved config enables it and a reranker model name is present.
- `services/config_service.py` now persists two new config fields:
  `enable_search_reranker` and `search_reranker_model`.
- `models/generation_job.py` now freezes those reranker settings into each
  request snapshot.
- `controllers/chat_controller.py` now copies reranker settings into the
  GenerationJob created at send time.
- `ui/settings_window.py` now exposes user-editable controls for enabling the
  dedicated reranker and selecting the reranker model name.
- The bundled unit tests were expanded to validate reranker ordering behavior,
  reranker-failure fallback behavior, and reranker-config propagation.

Validation update:
- Command run: `python -m unittest -v`
- Observed result: 47 tests passed.


Adversarial Review Reconciliation
---------------------------------
After this audit file was originally written, an external adversarial review
identified several real defects that the prior pass had not caught, most notably
the assistant/tool message-ordering issue in `services/ollama_client.py`. This
revision corrects those defects and updates the repository documentation so this
audit artifact is no longer claiming a cleaner state than the code actually had
at that time.


Live Pipeline Test Audit Note
-----------------------------
A new opt-in live integration test was added to `tests_web_search_integration.py`.
This test does not replace the existing mocked regression tests. Its role is
to make real SearXNG search behavior and real fetch outcomes observable through
trace logging when the reviewer explicitly enables it. That makes the live
search/fetch path inspectable without making the default offline test suite
fragile.
